7.3 Transition Rules (ตัวอย่าง)
ตัวอย่าง state transition ที่ service ต้อง enforce
Requester
•	Draft → Submitted
Procurement Review
•	Submitted → Under Procurement Review
•	Under Procurement Review → Returned for Revision
•	Under Procurement Review → Approved
•	Under Procurement Review → Rejected
Publish
•	Approved → RFP Published / RFQ Open
Vendor phase
•	RFP Published / RFQ Open → Vendor Response In Progress
•	Vendor Response In Progress → Bidding Open (optional)
•	Vendor Response In Progress → Under Evaluation (if no bidding)
•	Bidding Open → Bidding Closed
•	Bidding Closed → Under Evaluation
Evaluation
•	Under Evaluation → Pending Approval
Approval
•	Pending Approval → Returned from Approval
•	Pending Approval → Rejected
•	Pending Approval → Completed
Exceptional
•	almost any non-closed stage → Cancelled (policy-based)

7.4 Why State Machine Matters
ถ้าไม่ทำ state rules ชัดเจน จะเจอปัญหา:
•	ข้ามขั้นตอน
•	approve ตรงโดยไม่ evaluate
•	vendor submit หลัง deadline
•	procurement ปิดเคสโดยไม่มี approver
•	audit ไม่ครบ
ดังนั้นต้องมี:
•	workflow.service
•	transition validator
•	timeline append on every transition

5.1 auth
Responsibility
•	login / register / refresh / logout
•	issue JWT access token
•	store refresh token hash
•	validate token
Controllers
•	POST /auth/register
•	POST /auth/login
•	POST /auth/refresh
•	POST /auth/logout
•	GET /auth/me
Services
•	AuthService
•	TokenService
•	PasswordService
Notes
•	refresh token rotation
•	hash refresh token ใน DB
•	optional account lockout
________________________________________
5.2 users
Responsibility
•	manage internal users / vendor users
•	query user info
•	activate / deactivate user
Notes
สำหรับ production จริง แนะนำเก็บ:
•	role
•	property scope
•	department scope
•	active status
________________________________________
5.3 procurements
Responsibility
core business object ของระบบ
รองรับ:
•	create RFI /RFP / RFQ
•	save draft
•	submit request
•	review by procurement
•	return/reject/approve
•	publish
•	cancel
•	detail with status/timeline/owner
Controllers
•	GET /procurements
•	POST /procurements
•	GET /procurements/:id
•	PATCH /procurements/:id
•	POST /procurements/:id/submit
•	POST /procurements/:id/review/approve
•	POST /procurements/:id/review/return
•	POST /procurements/:id/review/reject
•	POST /procurements/:id/publish
•	POST /procurements/:id/cancel
Services
•	ProcurementsService → CRUD + orchestration
•	ProcurementWorkflowService → transition validator
•	ProcurementReviewService → review actions
•	ProcurementPublishService → publish logic
•	ProcurementValidationService → business validations
•	ProcurementQueryService → filter/sort/paginate
________________________________________
5.4 vendor-management
Responsibility
•	vendor master data
•	vendor CRUD
•	vendor status
•	company profile
________________________________________
5.5 vendor-invitations
Responsibility
•	invite vendors to procurement
•	accept / decline invitation
•	invitation_status tracking
•	vendor view: my invitations
Business Rules
•	vendor invitation ต้องผูก procurement
•	vendor ที่ไม่ได้ invite ห้าม submit
•	procurement phase ต้อง publish/open แล้วจึง invite ได้
________________________________________
5.6 rfq-submissions
Responsibility
•	vendor submit quotation / proposal
•	edit ก่อน deadline
•	lock หลัง deadline
•	attach documents
Business Rules
•	vendor ต้อง accept invitation ก่อน submit (ถ้า policy กำหนด)
•	submit ได้เฉพาะก่อน deadline
•	vendor เห็นเฉพาะ submission ของตัวเอง
________________________________________
5.7 ebidding
Responsibility
•	create multi-round bidding
•	open / close round
•	vendor submit bids
•	blind bidding data shaping
Business Rules
•	vendor เห็นเฉพาะ bid ตัวเอง
•	internal users เห็น anonymized comparison
•	submit ได้เฉพาะ round open
•	submit หลัง close ไม่ได้
________________________________________
5.8 evaluation
Responsibility
•	assign evaluators
•	ensure 1 lead evaluator
•	collect score + comments
•	calculate avg / variance / vote summary
•	lead evaluator recommendation
•	optional AI-assisted scoring
Business Rules
•	procurement หนึ่งรายการ ต้องมี 1 lead evaluator
•	evaluator เห็นเฉพาะ assigned case
•	consolidation ต้องเกิดหลัง review ครบตามเกณฑ์
________________________________________
5.9 approval
Responsibility
•	submit case for approval
•	approver inbox
•	approve / return / reject
•	final approval trail
Business Rules
•	ห้าม complete โดยไม่ผ่าน approver
•	approval ต้องเกิดหลัง evaluation / consolidation พร้อมแล้ว
________________________________________
5.10 results
Responsibility
•	announce winner
•	return vendor result view
•	close procurement
Business Rules
•	losing vendor เห็นแค่ Not Selected
•	vendor ห้ามเห็น ranking / competitor pricing / evaluator score
________________________________________
5.11 timeline
Responsibility
•	append timeline event for every critical workflow action

UI-oriented data
ใช้แสดงหน้า detail workspace และ audit timeline

11) Business Rule Enforcement Map
กฎสำคัญต้อง map เข้ากับ service ให้ชัด
________________________________________
Rule 1:
Requester create RFI, RFP and RFQ
→ ProcurementsService.create()
Rule 2:
All requests must pass Procurement Review before publishing
→ ProcurementPublishService.assertReviewApproved()
Rule 3:
Every case must track status, current_owner_role, full timeline history
→ ProcurementsRepository.updateStatus() + TimelineService.appendEvent()
Rule 4:
Multi-evaluator required: 1 lead + N evaluators
→ EvaluatorAssignmentsService.validateAssignments()
Rule 5:
Each evaluator submits score + comment; aggregate avg / vote count / variance
→ ScoreCalculationService.calculate(procurementId)
Rule 6:
Every case must go through Approver
→ ApprovalService.submitForApproval() and ResultsService.close() must assert approved
Rule 7:
Losing vendor sees only "Not Selected"
→ ResultVisibilityService.toVendorView()

4) SQL Indexes / Constraints ที่ควรเสริมด้วย Raw Migration
มีบาง business constraints ที่ Prisma schema อย่างเดียวบังคับไม่ได้ครบ
โดยเฉพาะ partial unique index และบาง workflow rule
ด้านล่างคือสิ่งที่ผมแนะนำให้เสริมใน migration SQL
________________________________________
4.1 Exactly one lead evaluator per procurement
Business rule: 1 procurement ต้องมี lead evaluator ได้เพียง 1 คน
Raw SQL
SQL
CREATE UNIQUE INDEX uq_evaluator_assignments_one_lead_per_procurement
ON evaluator_assignments (procurement_id)
WHERE is_lead = true;
Show more lines
หมายเหตุ:
Prisma ทำ @@unique([procurementId, isLead]) ไม่ได้ตาม intent จริง
เพราะมันจะบังคับทั้ง is_lead = false ด้วย ซึ่งไม่ต้องการ
________________________________________
4.2 Only one latest bid per vendor per round
SQL
CREATE UNIQUE INDEX uq_ebidding_responses_latest_per_vendor_round
ON ebidding_responses (ebidding_round_id, vendor_id)
WHERE is_latest = true;
Show more lines
________________________________________
4.3 Optional: only one active refresh token per device/session
ถ้าภายหลังต้องการ secure ขึ้นอีก สามารถเพิ่ม column เช่น device_id แล้วบังคับ unique ตาม user+device ได้
________________________________________
4.4 Optional check constraints
Prisma ยังไม่ครอบคลุม check constraint บางแบบดีพอ จึงแนะนำใส่ SQL เพิ่ม:
submission deadline sanity
SQL
ALTER TABLE procurements
ADD CONSTRAINT chk_procurements_deadline_after_created
CHECK (submission_deadline IS NULL OR submission_deadline > created_at);
Show more lines
bidding round time sanity
SQL
ALTER TABLE ebidding_rounds
ADD CONSTRAINT chk_ebidding_rounds_time
CHECK (ends_at > starts_at);
Show more lines
evaluation scores within 0-100
ถึงแม้ app layer จะ validate แล้ว แต่ DB-level check ก็ช่วยมาก
SQL
ALTER TABLE evaluator_reviews
ADD CONSTRAINT chk_evaluator_reviews_score_range
CHECK (score >= 0 AND score <= 100);

7.4 Why I added procurement_result_vendor_statuses

6. Recommended Design System
6.1 Recommended UI Approach
Material UI (MUI) เหมาะสุดสำหรับระบบ enterprise ลักษณะนี้ เพราะ:
•	component library ครบ
•	data-heavy screens ดี
•	form/table/dialog แข็งแรง
•	theming ง่าย
•	accessibility พร้อม
•	ทีม dev ดูแลง่ายกว่า custom-heavy Tailwind ตั้งแต่วันแรก
ถ้าองค์กรอยาก design freedom สูงมาก ค่อยใช้ Tailwind + headless components ภายหลัง
แต่สำหรับ enterprise procurement system, MUI คือ safe/best-fit choice

8. UX Notes and Next-Step Implementation Plan
8.1 UX Principles Applied
1) Workflow clarity
ผู้ใช้ต้องเห็นเสมอว่า:
•	ตอนนี้อยู่ stage ไหน
•	ใครเป็น current owner
•	next action คืออะไร
2) Minimal clutter
•	ใช้ section-based layout
•	ลด tab ซ้อน tab
•	ลด modal overload
3) Role isolation
•	vendor ไม่ควรเห็น internal complexity
•	evaluator ไม่ควรเจอ controls ฝั่ง procurement
•	procurer ไม่เสียเวลาไล่หาข้อมูลข้ามหน้า
4) Decision support
•	evaluation ต้องมี comparison + score + AI insight ในหน้าเดียว
•	approval/result ควรสรุปให้ผู้ตัดสินใจไว
5) Enterprise scalability
•	รองรับหลาย property / department
•	รองรับ table/filter data หนัก
•	role-based menus และ route guards ชัด

8.2 API Compatibility Notes (สำคัญ)
Frontend redesign นี้ยังคง compatible กับ backend ปัจจุบันได้ โดยยึด field สำคัญเดิม:
•	title
•	description
•	budget
•	invitation_status
และ flow เดิม:
•	POST /auth/login
•	GET /procurements
•	GET /procurements/:id
•	POST /vendor-invitations
•	GET /my-invitations
•	POST /rfq-submissions
•	POST /ebidding-responses
•	POST /calculate-scores
•	GET /procurement-results/:procurement_id

สร้าง client adapter layer เพื่อ normalize payload ระหว่าง frontend ใหม่กับ backend เดิม จะช่วยลด breaking changes มากที่สุด

8.5 Final Recommendation
สำหรับระบบนี้ ถ้าต้องการ enterprise-grade frontend ที่พร้อมต่อ production จริง ผมแนะนำสรุปดังนี้:
•	ใช้ React + TypeScript + MUI
1. Executive Summary
Frontend ใหม่ของระบบ Enterprise E Bidding Platform ควรเปลี่ยนจากโครงสร้างแบบ prototype/tab-heavy ไปเป็น role-aware, workflow-first enterprise application ที่เน้น:
•	Procurement clarity — ทีมจัดซื้อเห็นสถานะ, owner, next action ชัด
•	Vendor simplicity — vendor ส่งราคา / เข้าห้อง bid ได้ง่าย ไม่สับสน
•	Evaluator confidence — เห็น comparison, scoring, AI assist แบบตัดสินใจง่าย
•	Scalable architecture — พร้อมขยายหลาย property / department / role
•	Backend compatibility — ใช้กับ API ปัจจุบันได้โดยไม่ต้อง rewrite backend ก่อน
หลักการ redesign
•	ใช้ AppShell + Sidebar + Topbar + Workspace Layout
•	ใช้ role-specific dashboards
•	ใช้ detail workspaces แบบ section-based แทน tab กระจัดกระจาย
•	ใช้ consistent tables / forms / badges / timeline
•	แยก view model สำหรับแต่ละ role ชัดเจน
Recommended frontend stack
•	React + TypeScript
•	Material UI (MUI) สำหรับ enterprise UI
•	React Router
•	React Query / TanStack Query
•	React Hook Form + Zod
•	Axios
•	Context / Zustand สำหรับ auth + UI state
•	Day.js สำหรับ date formatting

6. Recommended Design System
6.1 Recommended UI Approach
Material UI (MUI) เหมาะสุดสำหรับระบบ enterprise ลักษณะนี้ เพราะ:
•	component library ครบ
•	data-heavy screens ดี
•	form/table/dialog แข็งแรง
•	theming ง่าย
•	accessibility พร้อม
•	ทีม dev ดูแลง่ายกว่า custom-heavy Tailwind ตั้งแต่วันแรก
ถ้าองค์กรอยาก design freedom สูงมาก ค่อยใช้ Tailwind + headless components ภายหลัง
แต่สำหรับ enterprise procurement system, MUI คือ safe/best-fit choice
________________________________________
6.2 Color Palette (Enterprise Procurement Theme)
Primary
•	#1E3A8A — Deep Enterprise Blue
•	#2563EB — Interactive Blue
Neutral
•	#0F172A — Heading / dark text
•	#334155 — secondary text
•	#64748B — muted text
•	#E2E8F0 — borders
•	#F8FAFC — page background
•	#FFFFFF — cards/surfaces
Semantic
•	Success: #16A34A
•	Warning: #F59E0B
•	Error: #DC2626
•	Info: #0EA5E9
________________________________________
6.3 Typography
•	Font: Inter หรือ Roboto
•	Headings: semi-bold / bold
•	Body: regular
•	Numeric fields: tabular alignment ถ้าใช้ตารางราคา
Scale
•	H1: 28
•	H2: 22
•	H3: 18
•	Body: 14–16
•	Caption: 12
________________________________________
6.4 Spacing & Layout
•	Base spacing: 8px
•	Card padding: 16px / 24px
•	Page section gaps: 24px
•	Border radius: 10px
•	Max content width: fluid with grid layout
•	Right rail width: 320–360px
________________________________________
6.5 Cards
•	white surface
•	soft shadow
•	subtle border
•	consistent padding
•	use cards for: 
o	summary
o	quick stats
o	timeline blocks
o	form sections

6.6 Tables

type AuthState = {
accessToken: string | null;
user: AuthUser | null;
setAuth: (token: string, user: AuthUser) => void;
logout: () => void;
};

export const useAuthStore = create<AuthState>((set) => ({
accessToken: localStorage.getItem('accessToken'),
user: localStorage.getItem('authUser')
? JSON.parse(localStorage.getItem('authUser') as string)
: null,
setAuth: (token, user) => {
localStorage.setItem('accessToken', token);
localStorage.setItem('authUser', JSON.stringify(user));
set({ accessToken: token, user });
},
logout: () => {
localStorage.removeItem('accessToken');
localStorage.removeItem('authUser');
set({ accessToken: null, user: null });
},
}));

import { useState } from 'react';
import { Box, Button, Card, CardContent, Stack, TextField, Typography } from '@mui/material';
import { useNavigate } from 'react-router-dom';
import { login } from '../services/auth.service';
import { useAuthStore } from '../store/auth.store';

export default function LoginPage() {
const navigate = useNavigate();
const setAuth = useAuthStore((s) => s.setAuth);

const [email, setEmail] = useState('procurement1@company.com');
const [password, setPassword] = useState('StrongPassword123!');
const [loading, setLoading] = useState(false);

async function handleSubmit(e: React.FormEvent) {
e.preventDefault();
setLoading(true);
try {
const result = await login({ email, password });
setAuth(result.accessToken, result.user);
navigate('/dashboard');
} finally {
setLoading(false);
}
}

return (
<Box sx={{ minHeight: '100vh', display: 'grid', placeItems: 'center', bgcolor: '#F8FAFC' }}>
<Card sx={{ width: 420 }}>
<CardContent sx={{ p: 4 }}>

<Typography variant="body2" color="text.secondary">
Secure access to E-Bidding platform
</Typography>
</Box>

<TextField
label="Email"
value={email}
onChange={(e) => setEmail(e.target.value)}
fullWidth
/>
<TextField
label="Password"

fullWidth
/>

13) Enterprise UX Notes
สิ่งที่ starter นี้ทำถูกสำหรับระบบ procurement
•	ใช้ workspace layout แทน tab-heavy MVP
•	procurement detail เป็น single source of truth
•	role-based menu ลด clutter
•	data tables พร้อมรองรับระบบจริง
•	AI อยู่ในตำแหน่ง decision support ไม่แย่ง control
สิ่งที่ควรทำต่อทันทีหลัง starter นี้
•	เพิ่ม React Query hooks ต่อ page
•	เพิ่ม loading / empty / error state
•	เพิ่ม dialog actions เช่น invite vendor / announce winner
•	เพิ่ม form validation ด้วย React Hook Form + Zod
•	เพิ่ม currency/date formatting utilities
•	เพิ่ม status mapping ให้ครอบคลุมทุก workflow state
